# 競馬予想アプリ (keiba-yosou)

## 案件概要
JRA中央競馬の予想を自動化する Streamlit Webアプリ。
個人利用(開発者の父親が日常使い)。商用配布は一切しない。
お父様の予想にかかる時間を、現状1〜2時間 → 5分に短縮することが目的。

## ターゲットユーザー
- 開発者の父親(Windows 10/11、ブラウザは Edge)
- JRA-VAN DataLab 契約者
- 競馬歴が長く、独自の予想ロジックを言語化できる
- PC操作は中級レベル(ブラウザ操作・ファイル管理は問題なし)

## 配布形態
Streamlit Community Cloud (無料プラン・公開リポジトリ)
- お父様はブラウザでURLアクセス → 当日出馬表CSVアップロード → 予想結果表示
- お父様PCには何もインストール不要

## 技術スタック
- 言語: Python 3.11+
- フレームワーク: Streamlit
- データ処理: pandas, numpy, pyarrow (Parquet用)
- 配布: Streamlit Cloud + GitHub 自動連携
- 過去データ形式: Parquet(GitHub リポジトリに同梱)
- 当日データ形式: CSV(お父様がアップロード)

## なぜこの構成か
- Streamlit Cloud: 無料、お父様PC設定不要、URLアクセスのみで利用可能
- Parquet: 1年分の過去データを CSV ではなく Parquet で持つことで GitHub の100MB制限内に収まる(120MB → 12MB)
- リポジトリ同梱: Streamlit Cloud はセッション間でアップロードファイルを保持しないため、固定データはリポジトリに含める

## 推奨馬選定ロジック(本ロジック v1.4 — rating-based + 直近10走評価)

> Phase 6 で v1.0 (○マーク方式、≥5 で ◎) から v1.1 (rating 合計、≥100 で ◎)
> へ移行済み。v1.3 で TARGET 指数を rating から除外した純粋ロジックモード化。
> **v1.4(2026-05 現在)**: ルール評価対象を直近 5 走 → 10 走に拡張し、
> ベテラン馬の長期実績を拾えるようにした。
> `prediction_logic.LOGIC_MODE = "rating"` がデフォルト。
> v1.0 の○マーク方式は `LOGIC_MODE = "onmark"` で互換維持。
> 仕様の単一情報源は `utils/rating_rules.py`(構造化データ)。

### 概要

予想は「直近10走の評価 → 各ルールの rate 加算 → ◎判定(rating ≥ 100)→
ワイド候補抽出 → 減点フィルタ → 買い目生成」の流れで構成される。
合計 rating で順位づけする数値型エキスパートシステム。

> **v1.4 注**: ルール評価対象は直近 10 走だが、**脚質判定**
> (`determine_running_style`)は引き続き直近 5 走の corner_1 平均で判定する
> (直近の脚質傾向が重要なため、10 走では古すぎる)。
> **直近 N 走戦歴マトリクス UI** も 5 走表示のまま(画面幅・お父様の慣れの都合)。

ルール体系(全 39 ルール、`utils/rating_rules.py`):
| カテゴリ | 件数 | rating 加算 | rate | 内容 |
|---|---|---|---|---|
| C | 14 | 〇 | 50 | 距離×馬場 別 上3F+通過順位改善 |
| D | 4 | 〇 | 20 | 距離無関係 上3F+通過順位改善 |
| E | 14 | 〇 | 20 | 距離×馬場 別 上3F のみ(通過順位改善 不要) |
| F1 | 1 | 〇 | 30 | ダート不良 + 逃げ脚質 |
| F2 | 1 | 〇 | 15 | 休養明け前走凡走 → 2,3走前で C/D/E 救済 |
| F3 | 1 | 〇 | 20 | 1600m以上 + 斤量 -3kg(前走比) |
| F4 | 1 | 〇 | 30 | 坂路調教 1F(Lap1) ≤ 11.2(v1.5 で実装) |
| F5 | 1 | 〇 | 40 | 坂路調教 1F + 2F ≤ 11.2(F4 排他、F5 が +40)|
| A1 | 1 | × | 2 | 1番人気枠の偶奇でワイド候補絞り込み |
| A2-A5 | 4 | × | 4-30 | ワイド候補フラグ + priority weight |
| B1 | 1 | × | 25 | 1番人気の逃げ → 2着以下扱い(◎除外) |
| B2 | 1 | × | 15 | 阪神1600m + 7,8枠 → 3着以下扱い |

集計ポリシー(`RatingPolicy.STRICT`、デフォルト):
- C/D/E は同一過去走で複数該当 → **最高 rate のみ採用**(over-counting 防止)
- 同一 rule_id が複数走で発火 → **1 回まで**(dedup)

判定:
- 合計 rating **≥ 100** で ◎本命確定。複数頭時は人気上位タイブレーク
- 100 点未満で◎不在 → 最高 rating 馬を **準◎** にフォールバック
- B1/B2 該当馬は ◎ 候補から除外(rating には影響しない)

### DC 形式サポート(v1.3 純粋ロジックモード)— TARGET ダイレクト出力 + historical 連携

> **v1.3(2026-05 現在の本番動作)**: お父様独自ロジック(C/D/E/F1/F2/F3)
> **だけ** で rating を計算する純粋ロジックモードに移行。TARGET 指数(ZI =
> col[5])は **参考値** として UI に表示するが、rating には一切含めない。
> マッチ失敗馬は rate=0 で「評価不能」扱いとして除外する。
> ◎本命閾値は RA+SE と同じ rating ≥ 100 に統一。
>
> **動機**: お父様が「自分のロジックだけで何位くらいの馬が浮上するか」を
> 純粋に評価したいと希望。v1.2 のハイブリッドでは rate 256 のうち約半分が
> TARGET 指数由来で、独自ロジックの実力が見えなかった。
>
> **v1.2(廃止済み、retain-only)**: フルモード = C/D/E/F + TARGET 指数加算、
> 簡易モード(マッチ失敗) = TARGET 指数 単独採用。閾値は DC のみ 200。
> v1.3 で完全に置き換えた。
>
> **共通の枠組み(v1.2 / v1.3)**: TARGET frontier JV の **DC(ダイレクト)系
> メニュー** から出力された CSV を直接受け付け、過去走パターンマッチで
> historical/races.parquet と紐付けることで、馬名・騎手等を持たない指数特化
> フォーマットでも本ロジック v1.1 をフル発火させる。

#### DC 形式の公式列定義(45 列、JRA-VAN「出馬表CSV形式」)

当日レース行(0-indexed):
- `col[0]`: 10桁レース ID(JRA場 2桁 + 年下2桁 + 開催コード 2桁 + R番 2桁 + 馬番 2桁)
- `col[1]`: クラスコード
- `col[2]`: 距離(m)
- `col[3]`: トラックコード(0=芝 / 1=ダ / 2=障害)
- `col[4]`: 頭数
- `col[5]`: ZI(TARGET 総合指数)
- `col[6]`: **前走との間隔(週)** — 当日 → 前走の週数
- `col[7]`: 確定着順(当日未走で常に 0)
- `col[8]`: 異常コード
- `col[9]`: 人気(morning オッズ取り込み後、未取り込み時は 0)

過去走 5 列(base = 10 + run_i × 5):
- `base+0`: クラスコード(class_code)
- `base+1`: 距離(distance)
- `base+2`: コースコード(course_code、surface に縮約運用)
- `base+3`: **前走との間隔(週)** — このランの前のラン(より新しい)からの週数
- `base+4`: 補正タイム(adjusted_time、0=記録なし)

⚠️ **過去走に「着順」は含まれない**(公式仕様)。
過去には誤って `col[base+3]` を `finishing_position` と解釈する偽陽性バグが
存在したが、commit `xxx` で公式仕様準拠に修正済み。

#### 過去走パターンマッチ(`utils/horse_matcher.py`)
- 各過去走の **推定 race_date** を `col[6]`(today の weeks_since_prior)を
  起点に weeks_since_prior チェーンを遡って計算:
  - `run[0].date = today_date - col[6] × 7 日`
  - `run[i+1].date = run[i].date - run[i].weeks_since_prior × 7 日`
- historical/races.parquet と **(race_date ± 3 日, distance ±0m)** で集合演算
- 各 past_run でヒットした horse_id 集合を横断的に「票」として集計
- **5 走以上で同一 horse_id にヒット** + 同票単独 → 採用(同票複数は不採用)
- マッチ成功率(実測 DC260509、仕様準拠版): **58.4% (289/495)** — 京都11R 8/16
  (旧着順誤読版の 73.1% は偽陽性混入だった)

#### モード分岐(v1.3 純粋ロジック)
- **マッチ成功(高/中信頼度)**:C/D/E ルールが上3F・通過順位・馬場で発火、
  F1/F2 も評価可能。**rate = ルール加算合計のみ**(TARGET 指数は不加算)。
  TARGET 指数は `HorseRating.target_index` に参考値として保持し UI 表示。
- **マッチ失敗**:過去走 DB に紐付けできない → **rate = 0 で「評価不能」**。
  サイドバーの推奨馬一覧(rating ≥ 100 フィルタ)から自然に除外され、
  全頭詳細セクションでは warning「🛇 評価不能(参考 TARGET 指数 X は無視)」
  を表示する。

#### 取得不能項目と影響
| 項目 | 取得経路 | 取れない場合のルール影響 |
|---|---|---|
| 馬名 / 騎手 | historical 引当 | マッチ失敗で「馬番N」表示 |
| 上3F | historical の last_3f | C/D/E ルールが発火不能 |
| 通過順位 | historical の corner_1..4 | C/D 発火不能、脚質判定不可 |
| 当日馬場 | UI ラジオで入力 | 未入力時は「良」デフォルト |
| 当日斤量 | DC に無い | F3 (1600m+ 斤量-3kg) 永続無効 |
| 単勝オッズ | DC に無い | TARGET 指数順を「人気」として代替 |

#### 当日馬場入力 UI
DC 形式検知時のみサイドバーに「🏟 当日馬場」ラジオ表示。
選択肢: 良 / 稍重 / 重 / 不良。session_state["dc_going"] に保存し、
レース絞り込み・予想実行を跨いで維持。

### 必要データ

各馬の現在のレース情報に加え、以下を取得・計算:

| データ | 取得元 | 備考 |
|---|---|---|
| 着順 | historical | 過去戦歴の各レース |
| 上がり3F | historical | 過去戦歴の各レース |
| 通過順位(各コーナー) | historical or raw CSV 要調査 | 1〜4コーナーの位置順 |
| 距離・芝/ダ・馬場・場 | race_card | 当日レース情報 |
| 馬番 → 枠番 | 計算 | 出走頭数から導出(8枠制) |
| 人気 | race_card | 単勝オッズ順 |
| 脚質 | 過去走から推定 | 過去5走の通過順位パターンから判定 |
| 休養明け判定 | 過去走の日付差 | 前走から180日以上空いていれば休養明け |

### 脚質の判定基準

過去5走の初角(1コーナー)通過順位の平均で判定:
- 平均1〜3番手  → 逃げ
- 平均4〜6番手  → 先行
- 平均7〜10番手 → 差し
- 平均11番手以下 → 追込
- 過去走3走未満 → 不明(暫定で先行扱い)

### Step 1: ○マーク収集ルール群

各馬について以下14ルールを評価し、該当ごとに○を1個付与する。
○が5個以上付いた馬は ◎候補 となる。

#### 上がり3F + 通過順位ルール(過去走で評価、各ルール1回まで○加算)

「通過順位が小さくなっている」= レース後半のコーナーほど位置順が上がっている
(例: 1コーナー10番手 → 2コーナー8番手 → 3コーナー5番手 → 4コーナー3番手)

【芝・短距離】
- ルール9: 芝1400m以下・良馬場 + 上がり3F < 33.3秒 + 通過順位改善
  - 阪神・中山は閾値 33.5秒 以内
- ルール10: 芝1400m以下・重馬場 + 上がり3F < 34.0秒 + 通過順位改善
  - 阪神・中山は閾値 34.2秒 以内

【芝・マイル】
- ルール11: 芝1600m・良馬場 + 上がり3F < 34.2秒 + 通過順位改善
- ルール12: 芝1600m・重馬場 + 上がり3F < 35.0秒

【芝・中距離】
- ルール13: 芝1800〜2000m・良馬場 + 上がり3F < 34.0秒 + 通過順位改善
  - 阪神・中山は閾値 34.5秒 以内
- ルール14: 芝1800〜2000m・重馬場 + 上がり3F < 35.0秒 + 通過順位改善
  - 阪神・中山は閾値 35.5秒 以内

【芝・長距離】
- ルール15: 芝2200m以上・良馬場 + 上がり3F < 35.0秒 + 通過順位改善
  - 阪神・中山は閾値 35.5秒 以内
- ルール16: 芝2200m以上・重馬場 + 上がり3F < 35.5秒 + 通過順位改善
  - 阪神・中山は閾値 36.0秒 以内

【ダート・短距離】
- ルール17: ダート1400m以下・良馬場 + 上がり3F < 35.0秒 + 通過順位改善
- ルール18: ダート1400m以下・重馬場 + 上がり3F < 36.0秒 + 通過順位改善

【ダート・中距離】
- ルール19: ダート1600〜2000m・良馬場 + 上がり3F < 36.0秒 + 通過順位改善
- ルール20: ダート1600〜2000m・重馬場 + 上がり3F < 35.5秒 + 通過順位改善

【ダート・長距離】
- ルール21: ダート2200m以上・良馬場 + 上がり3F < 37.0秒 + 通過順位改善
- ルール22: ダート2200m以上・重馬場 + 上がり3F < 36.5秒 + 通過順位改善

#### 休養明け関連ルール

- ルール24: 休養明け処理
  - 「休養明け」 = 前走から 180日以上 空いている
  - 休養明け前走が5着以下の場合、2走前・3走前 のレースを評価対象として
    通常の上がり3Fルール(9〜22)を適用し、該当すれば○を加算
  - つまり休養明けで前走凡走の馬は、前走を無視して2,3走前で評価する救済ルール

### Step 2: ◎本命判定

- ○マーク数が 5個以上 の馬を ◎(本命) と判定
- 該当馬が複数いる場合は人気・タイム指数等でタイブレーク
- 該当馬がいない場合は ◎なし(後述の減点・補正後に再判定)

### Step 3: 減点・除外ルール

以下に該当する馬は ◎としては選ばない(2着以下扱い、または3着以下扱い)

- ルール6: 単勝1番人気 かつ 逃げ脚質 → 2着以下扱い
  (狙われやすいので軸馬には不適、ただし3着候補にはなり得る)
- ルール7: 阪神1600m + 外枠(7枠 or 8枠) → 3着以下扱い

### Step 4: ワイド候補抽出(◎とは別軸、最大3頭まで)

以下のルールに該当する馬から最大3頭までをワイド候補としてリストアップ:

- ルール3: 「単勝1番人気の馬」の隣枠 + 7番人気以降 → ワイド候補
- ルール4: 前走着順が 5・7・9・11・13着 → ワイド候補
- ルール5: 1枠の逃げ馬 + 5番人気以降 → ワイド候補
- ルール8: 競馬場が小倉 or 中京 + 逃げ脚質 → ワイド候補

複数該当する馬は優先度高として上位に。3頭を超える場合は人気上位を優先。

### Step 5: 買い目戦略・補正

- ルール2: 「単勝1番人気の馬」の枠の偶奇に応じて穴の買い目絞り込み
  - 1番人気が奇数枠(1,3,5,7) → 穴の買い目候補は奇数枠の馬から
  - 1番人気が偶数枠(2,4,6,8) → 穴の買い目候補は偶数枠の馬から
- ルール23: ダート + 不良馬場 → 逃げ脚質の馬に補正(プラス評価、本命候補に格上げ)

### 推奨買い目の出力(全券種)

判定結果に基づき、以下の全券種の推奨買い目を生成:

- 単勝: ◎の馬
- 複勝: ◎の馬
- 馬連: ◎ — ワイド候補上位
- 三連複: ◎ — ワイド候補上位2頭(BOX or 流し)
- ワイド: ◎ — ワイド候補3頭(全組み合わせ)

### UI 表示要件

- 各馬の○マーク数と該当ルール一覧を表示
- ◎本命 / 注目馬(○3〜4個) / その他 / ワイド候補 / 危険人気馬 を別セクションで表示
- 推奨買い目を券種別に表示
- ロジック透明性のため「なぜこの馬が◎か」のルール根拠を展開可能に

### 本ロジック実装の前提確認事項
- raw CSV に通過順位(各コーナー)列が存在するか確認が必要
  (TARGET 52列のうち [27]-[31] あたりが候補になりそうだが、未検証)
- 存在しない場合は raw データの再エクスポート要請が必要
  (TARGET frontier JV のメニュー設定変更で「コーナー通過順」を含める)
- 脚質判定は過去5走の初角通過順位の平均で算出
- 枠番は 8枠制で出走頭数から計算(1-8頭=馬番=枠番、9-16頭は2頭ずつ、18頭=2+2+2+2+2+2+3+3 等)

### v1.0 運用後のチューニング候補(Phase 5 完了後の課題)
本ロジック v1.0 は Phase 5 で完全統合済み(prediction_logic.py + app.py UI)。
実運用で観察された傾向:
- **○ ≥ 5 の本命候補が長距離G1で出にくい**: 京都11R 天皇賞春G1(15頭)では
  最高 ○2(タガノデュード)止まり。spec の「該当なし→準◎」の fallback
  で運用は成立するが、お父様への運用提案として **○ 閾値を 3〜4 に下げる**
  選択肢を utils/judgment_engine.HONMEI_MARK_THRESHOLD で調整可能に
  しておく(現状 = 5)。
- **odds 列の解釈**: morning CSV の column [51] は単勝最終オッズではなく
  別指標(全頭が 50〜60 台に分布)である可能性。実 JRA 確定オッズが
  raw データに含まれていないか、別列(現在未マッピング)に格納されている
  可能性があるため、Phase 6 で再調査の余地。実 1 番人気と算出 1 番人気
  に乖離あり。
- **稍重馬場の扱い**: spec が「良 / 重」二分のため稍重はどのルールにも
  該当しないが、稍重 = 良寄り or 重寄りどちらに分類するかは
  お父様の経験値で確定する余地あり。

## エラーハンドリング方針
- 想定外のCSV列構成: 日本語で「列名が想定と異なります」と表示し期待する列名一覧を提示
- データ不足の馬: 「過去データなし」と表示してスキップ
- ファイル読み込み失敗: 日本語エラーメッセージ
- 予想ロジック内例外: 該当馬をスキップして処理継続

  ただし placeholder は Streamlit のメイン描画領域に属するため、rerun の
  暗転中は overlay が消えており、**描画完了直前 1 秒だけしか馬が見えない**
  問題があった(v1.6.1 で MutationObserver による検知も試みたが、新 DOM
  注入後にしか発火せず根本解決にならず)。
- **v1.6.2 UX**(2026-05): overlay を CSS-first(`body:has(...)`)に書換え。
  ただしページロード時や何でもない瞬間に overlay が常時表示される致命
  バグ → v1.6.3 で撤回。
- **v1.6.3 UX**(2026-05): ホワイトリスト方式(予想実行ボタン + 競馬場
  ラジオ click 検出)に変更。常時表示は解消したが、click event listener
  が React の `stopPropagation()` で発火しない / 即座に hide される問題で
  実機では「馬が一切表示されない」状態に。
- **v1.6.4 UX**(2026-05): **状態機械 + 最小表示時間 + 多重イベント
  + capture phase + Python 側フォールバック** で再構築。
  * 状態機械 `IDLE → SHOWN → MIN_TIME_PASSED → RUNNING_DETECTED → IDLE`
    で **最低 800ms** は必ず表示(点滅して見えない問題回避)
  * `running` を観測したら `running` 終了まで hide しない、観測なしなら
    5 秒で見切り hide、暴走対策に 30 秒 safety hide
  * `click` / `change` / `pointerdown` を **capture phase** で attach
    (React の stopPropagation を回避)
  * ボタンテキスト判定は `replace(/\s+/g, '')` で「予想実行」マッチ
    (emoji 🎯 や全角空白に対応)
  * ラジオは `<input type="radio">` / `<label>` 親 / `[role="radio"]` /
    `[data-baseweb="radio"]` を全網羅、競馬場かは親 group text に
    「競馬場」を含むかで判定
  * `iframe` 両対応(`parent.document` を try → 失敗時 `document`)
  * `console.log('[HorseOverlay]', ...)` で各段階を出力する DEBUG モード
    搭載(`render_running_horse_overlay(debug=True)` で有効化)
  * Python 側 `trigger_overlay_inline()` で予想実行ボタン handler から
    `window.__showHorseOverlay()` を即時呼び出して保険。状態機械の早期
    return で二重発火しても無害
  * 注入合計 12.1 KB、`:has()` 不使用で常時表示バグの再発防止
- **v1.6.5 UX**(現行、2026-05): **診断版 — `components.html()` 経路 +
  可視デバッグバッジ + 強制トリガー UI**。
  * 主因仮説の確定: Streamlit 1.30+(本プロジェクト 1.57.0)では
    `st.markdown(unsafe_allow_html=True)` 経由の `<script>` がサニタイザで
    実行されない。v1.6.0〜v1.6.4 が実機で機能しなかった根本原因。
  * 解決: `streamlit.components.v1.html(html, height=0, width=0)` で 0px
    の不可視 iframe を作り、その内部で実行された `<script>` から
    `window.parent.document` に overlay を注入する公式保証経路に切替。
  * 画面右下に **`#horse-debug-badge`** を常時表示し、JS の各段階で
    textContent を更新(`overlay: ready (parent)` / `click@btn:予想実行`
    / `SHOWN(predict)` / `RUNNING_DETECTED` / `IDLE` 等)。
    お父様が DevTools を開かなくても状態判別可能。
  * インストール時に環境調査ログを強制出力(iframe 検出 / CSP テスト /
    DOM 要素カウント for button・input[type=radio]・[role=radio]・
    [data-baseweb=radio])。
  * 5 階層 textContent 検査でラジオ判定を仮想スクロール対応に。
  * サイドバーに **🔧 診断モード** チェックボックスを設け、
    「🐎 馬を強制表示テスト」+「📡 listener 状況確認」ボタンで原因特定可能。
  * 強制表示で馬が出る → イベント検知の問題、出ない → CSS / overlay 注入
    の問題、と切り分け可能。
  * DEBUG=True デフォルト、原因特定後 v1.7 で診断 UI と DEBUG を撤去予定。

【お父様向け診断 6 ステップ(v1.6.5)】
1. アプリを開く → 画面 **右下** に黒い小さなバッジ `overlay: ready (...)`
   が見えるか確認
   - 見えない → JS 自体が動いていない(CSP / iframe sandbox 等)
   - 見える → JS 注入成功、次へ
2. サイドバーの「🔧 診断モード」をチェック
3. 「🐎 馬を強制表示テスト」ボタンを押す
   - 馬が画面を走る → 注入は OK、残るは「click イベントが届いていない」
   - 馬が出ない + alert が出る → JS 経路は生きてるが overlay show 失敗
   - 何も出ない → JS 全体が止まっている
4. 「📡 listener 状況確認」を押す → alert で `installed=true` を確認
5. 「予想実行」ボタンや競馬場ラジオを押した時、右下バッジが
   `click@btn:予想実行` や `click@radio:競馬場(click)` に変わるか確認
   - 変わる + `SHOWN` に進む → 馬が出ているはず
   - 変わらない → React で stopPropagation されている可能性
6. バッジの最終状態 + 馬が出たか + alert の内容を開発者に報告
